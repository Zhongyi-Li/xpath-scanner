# iframe 页面元素扫描设计

## 背景与根因

当前扫描器仅在主页面执行一次 `page.evaluate()`。投诉管理页面中的“投诉单查询”表单位于跨域 iframe：主页面只有 2 个 input，而业务 iframe 中有 12 个 input，因此订单编号、投诉编号、日期、投诉原因、投诉状态等元素没有进入候选列表。

## 目标

1. 每次用户输入 `scan` 时，扫描当前主页面以及其中所有可见、已加载的业务 iframe。
2. 保持交互式扫描方式，不自动切换 Tab、不自动触发业务操作。
3. iframe 内的元素继续写入既有 7 列结构，不新增 Sheet 或列。
4. 定位方式能够同时表达 iframe 入口和 iframe 内元素，避免只能在错误的文档上下文中执行 XPath。

## 方案

### 扫描范围

先在主 frame 采集页面层级和主页面元素，再遍历 Playwright 提供的子 frame。仅采集满足以下条件的 frame：

- 对应 iframe 元素可见；
- frame 已加载并可读取 DOM；
- 未命中登录、验证码、风控、报告页等禁止规则；
- 存在可见、可操作的候选元素。

预加载脚本 frame、空白 frame 和隐藏 frame 不写入结果。

### 页面路径与语义

iframe 内元素继承主页面已经识别出的页面层级和当前 Tab，例如：

`交易 > 投诉与申诉 > 我被投诉 > 违约投诉管理 > 订单编号`

元素名称、元素类型、日期区间命名、Radio 状态路径等规则沿用现有通用语义逻辑，不添加“投诉管理”等业务名称硬编码。

### 定位方式

主页面元素继续使用原 XPath。iframe 内元素在同一个“定位方式”单元格中使用组合格式：

`frame=<iframe XPath> >>> xpath=<控件 XPath>`

该格式明确两个执行步骤：先定位 iframe，再进入该 frame 查找控件。iframe XPath 优先使用稳定的 `src`、`title`、`name` 等属性，避免动态 id 和随机 class。

### 去重与落盘

唯一键仍为“页面路径 / 元素名称 / 元素类型 / 适用流程”。组合定位方式参与现有 XPath 稳定性比较；相同页面状态再次扫描时跳过，切换 Tab 后出现的新 iframe 控件正常新增。每次扫描仍按 JSON、进度 JSON、Excel 的顺序实时落盘。

## 错误处理

- 单个 frame 因跳转或卸载而读取失败时，记录到 `xpath-debug.log`，继续扫描其他 frame。
- Excel 写入失败时继续保证两个 JSON 文件已保存。
- iframe 命中禁止页面规则时跳过，不读取其元素。

## 测试与验收

1. 单元测试先覆盖 iframe 组合定位格式和禁止 frame 过滤，并确认测试在实现前失败。
2. 运行全部单元测试和 TypeScript 类型检查。
3. 连接当前本地 Chrome，只读验证投诉管理页面：主页面元素与业务 iframe 中的 12 个 input 均能进入扫描候选。
4. 实际执行一次临时扫描，确认表单元素拥有“我被投诉”页面路径和可执行的 frame + XPath 组合定位。
5. 确认输出仍只有 `XPath清单` Sheet 和固定 7 列。
